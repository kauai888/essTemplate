üìå Key Relationships

Users ‚Üí Attendance: One employee has many attendance records. One-to-Many relationship via user_id.
Users ‚Üí Leave Requests: One employee can submit multiple leave requests. One-to-Many relationship via user_id.
Leave Requests ‚Üí Approvals: Each leave request triggers an approval record. One-to-One relationship via request_id.
Users ‚Üí Leave Balance: One employee has leave balances for each type per year. One-to-Many relationship via user_id.
Users ‚Üí Credentials: One employee can have multiple credentials. One-to-Many relationship via user_id.
Users ‚Üí Expense Claims: One employee can submit multiple expense claims. One-to-Many relationship via user_id.
Expense Claims ‚Üí Approvals: Each expense claim requires approval. One-to-One relationship via request_id.
Users ‚Üí Users (Manager): Manager relationship through manager_id in Users table. Hierarchical self-referencing.
Approvals ‚Üí Users: Approvals link both requestor and approver to Users. One-to-Many for both directions.
Audit Log ‚Üí All Tables: Tracks all changes across system. Records user_id, table_name, record_id, and JSON details.
